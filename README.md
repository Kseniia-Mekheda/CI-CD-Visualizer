# CI/CD Visualizer (Pipely)

Вебзастосунок для **візуалізації CI/CD пайплайнів** з YAML-файлів (наприклад, GitLab CI): залежності між `jobs` будуються у вигляді інтерактивного графа. Для зареєстрованих користувачів доступні **збереження історії**, **редагування YAML** та **AI-аналіз** конфігурації.

## Мета проєкту

Спростити розуміння складних пайплайнів: замість читання довгого YAML одразу видно, які джоби від чого залежать (`needs`), можна переглянути деталі кожної джоби та отримати підказки щодо безпеки, продуктивності та кращих практик.

## Функціонал

| Можливість | Опис |
|------------|------|
| **Завантаження YAML** | Підтримуються файли `.yaml` / `.yml`; парсинг і побудова графа на бекенді. |
| **Граф залежностей** | Вузли — джоби, ребра — зв’язки з `needs` (React Flow). |
| **Без акаунта** | Можна завантажити файл і подивитися граф; історія не зберігається. |
| **Реєстрація / вхід** | Email і пароль або Google OAuth. |
| **Історія конфігурацій** | Після входу завантаження зберігається в PostgreSQL; можна відкрити, оновити або видалити запис. |
| **Режими на дашборді** | Перемикання між переглядом графа та коду (Monaco Editor). |
| **AI-звіт** | За збереженою конфігурацією — аналіз через Google Gemini (категорії: безпека, продуктивність, best practices); мова звіту — українська або англійська. |
| **Локалізація** | Інтерфейс українською та англійською. |

**Стек:** React (Vite, TypeScript), Zustand, Tailwind; FastAPI, SQLAlchemy, Alembic, PostgreSQL; Docker Compose.

## Запуск через Docker

### Вимоги

- [Docker](https://docs.docker.com/get-docker/) та Docker Compose (v2).

### 1. Змінні для бази даних (корінь репозиторію)

Скопіюйте шаблон у `.env` і заповніть значення (у PowerShell: `Copy-Item .env.template .env`; у bash: `cp .env.template .env`).

У файлі `.env` у корені проєкту задайте змінні `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` — їх використовує сервіс `db` у `docker-compose.yml` і підключення бекенду.

### 2. Змінні бекенду

Скопіюйте шаблон у `backend/.env` і заповніть усі поля з `backend/.env.template`:

- `SECRET_KEY`, `ALGORITHM`, `ACCESS_TOKEN_EXPIRE_MINUTES`, `REFRESH_TOKEN_EXPIRE_DAYS`
- `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI` (для OAuth)
- `GEMINI_API_KEY` (для AI-аналізу; без ключа відповідні функції не працюватимуть)

Під час запуску через Compose змінна `DATABASE_URL` для бекенду **перевизначається** автоматично згідно з сервісом `db`.

### 3. URL API для фронтенду (обов’язково перед збіркою)

У Vite адреса бекенду підставляється на етапі збірки. Створіть файл `client/.env` (або скопіюйте з `client/.env.local.template`) і вкажіть бекенд, доступний з браузера на вашій машині:

```env
VITE_API_URL=http://localhost:8000
```

Після зміни `VITE_API_URL` потрібна **повторна збірка** образу клієнта.

### 4. Збірка та старт

З кореня репозиторію:

```bash
docker compose build
docker compose up -d
```

Перший старт бекенду виконує `alembic upgrade head`, потім піднімається Uvicorn.

### 5. Доступ до сервісів

| Сервіс | Адреса |
|--------|--------|
| Фронтенд | http://localhost:3000 |
| API (Swagger за замовчуванням у FastAPI) | http://localhost:8000/docs |
| PostgreSQL (з хоста) | `localhost:5433` (порт змаповано з 5432 у контейнері) |
| Перевірка бекенду | http://localhost:8000/api/health |

### Зупинка та дані

```bash
docker compose down
```

Том `postgres_data` зберігає дані БД між перезапусками. Щоб повністю скинути БД, видаліть том: `docker compose down -v`.

### Примітки щодо OAuth та CORS

- У `backend/app/core/config.py` для CORS задані походження `http://localhost:3000` та `http://127.0.0.1:3000` — вони відповідають мапінгу порту клієнта в Compose.
- `GOOGLE_REDIRECT_URI` має збігатися з налаштуваннями OAuth-клієнта в Google Cloud Console.
